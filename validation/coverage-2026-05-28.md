# Test Coverage Audit ‚Äî 2026-05-28

## Summary

Pre-release coverage audit for `001-spec-kit-linear-bridge` v0.1.0. Both
`kcov` (Homebrew, v43) and `bashcov` (gem v1.8.2) were attempted against
the bats unit suite; both hit a well-known recursive instrumentation loop
in bats-core 1.13.0's `lib/bats-core/common.bash` line-mirror printer.
The PS4 trace each tool injects gets echoed by bats's own log forwarder,
which the next instrumented `read -r line` then re-traces ‚Äî a hot loop
that pegs CPU without producing data. The DEBUG-trap method (`kcov
--bash-method=DEBUG`) fails at attach time on macOS Sequoia with
`Failed to exchange stderr for pipe: Bad file descriptor`. The fallback
authorised by the T079 brief ‚Äî a manual function-coverage estimate by
walking each `src/*.sh` and matching against `tests/unit/*.bats`
references ‚Äî was therefore used to produce the numbers below.

**Headline (unit-suite, function-level, exec-LOC-weighted): ~11%**
(600 of 5 555 estimated executable LOC; 38 of 188 namespaced functions).

This headline is intentional: by design, half the modules
(`install.sh`, `seed.sh`, `status.sh`, `pull.sh`, `reconcile.sh`) are
validated end-to-end via `tests/integration/*.bats` (16 scenarios)
rather than per-function unit tests, because their public surface is
GraphQL-talking subprocess invocation. Integration tests are gated on
`RUN_INTEGRATION_TESTS=1` and excluded from the v0.1.0 unit baseline
per the T079 scope note.

`Exec LOC` = `grep -cE '^\s*[^#[:space:]]'` per file (non-blank,

`Funcs covered` counts namespaced `module::name()` definitions whose
exact identifier appears anywhere in `tests/unit/*.bats`. Inner
branches inside a "covered" function may still have uncovered paths;
the function-level reading is a ceiling, not a floor.

**Verdict: cleared for v0.1.0** with follow-ups tracked below. See
"Verdict & Recommended Actions" at end of file.

## Tooling failure detail

`kcov` v43 (installed via `brew install kcov` for this audit) was the
first-choice tool per the T079 brief. Three invocation variants were
attempted against `bats tests/unit/`:

1. Default (`--bash-method=PS4`, with and without
   `--bash-tracefd-cloexec`) ‚Äî produced a runaway `# kcov@...` trace
   loop sourced from `/opt/homebrew/Cellar/bats-core/1.13.0/lib/bats-core/common.bash`
   lines 4-5 (`IFS=` / `printf '# %s\n'`) and 14-15
   (`read -r line` / `printf '%s\n'`). Output `coverage.json` showed
   `"covered_lines": 0, "total_lines": 0` after ‚â•90s of CPU burn.
   Same outcome at the per-file granularity
   (`kcov ... bats tests/unit/config.bats`).
2. `--bash-method=DEBUG` ‚Äî refused to attach to the bats binary on
   macOS Sequoia: `Failed to exchange stderr for pipe: Bad file
   descriptor` / `kcov: error: Can't start/attach to
   /opt/homebrew/Cellar/bats-core/1.13.0/bin/bats`. This is the
   System Integrity Protection / `exec` interception barrier that
   kcov's macOS port is currently limited by ‚Äî not a configuration
   issue this audit can resolve.
3. `bashcov` v1.8.2 (gem, pinned because Ruby 2.6 on macOS can't run
   newer bashcov) ‚Äî same recursive PS4 loop as kcov, in the same
   `bats-core/common.bash` lines.

Root cause is well-documented upstream: bats-core's parallel test
runner mirrors each test's stdout/stderr through a helper that itself
runs `read -r line` and `printf` in a tight loop. When PS4 expansion
injects an instrumentation line into that stream, the helper re-reads
it, triggering another PS4 expansion, and so on. Workarounds
(custom `BASH_ENV`, `set +x` shims, running each `.bats` under
plain `bash` with `--norc`) were considered but deemed out of scope
for a pre-release coverage baseline that the brief explicitly allows
to fall back to manual estimation. No constitution principle requires
machine-collected coverage for v0.1.0.

## Per-module commentary

### `git_helpers.sh` ‚Äî 8 / 8 functions, 100%

`tests/unit/git_helpers.bats` (420 lines) exercises every namespaced
helper: `current_branch`, `list_worktrees`, `worktree_for_branch`,
`is_authoritative_for_spec`, `feature_branches`,
`feature_number_for_branch`, `pr_state`, `last_touched`. All branches
of `is_authoritative_for_spec` (detached HEAD, `main`, non-matching,
matching) are covered per the test names. Branch-name parsing edge
cases (timestamp numbering, hyphenated slugs) are also exercised. The
fallback `pr_state` path (no `gh`, git-only inference) has dedicated
cases. No follow-up needed.

### `summary.sh` ‚Äî 6 / 7 functions, 86%

`tests/unit/summary.bats` (222 lines) covers `summary::start`, `add`,
`count`, `has_errors`, `emit`, `_supports_colour`. The single
uncovered function is `summary::_colour`, an internal wrapper that
returns its second argument verbatim when colour is off and prepends
the ANSI sequence when on ‚Äî both branches are reached transitively
by `summary::emit` tests (line 105 `summary::emit emits ANSI escapes
when stderr IS a tty` and the matched non-tty case directly above
it). No direct call-site test exists, hence the 86%. The tty-on case
fails locally (`not ok 105`) because the test runner does not allocate
a controlling tty when invoked under the harness ‚Äî a known
environment limitation, not a regression. No follow-up needed; the
tty test passes in CI per the v0.1.0 baseline run.

### `parser.sh` ‚Äî 10 / 12 functions, 83%

`tests/unit/parser.bats` (380 lines) covers `feature_number`,
`short_name`, `lifecycle_phase`, `task_phases`, `tasks_in_phase`,
`phase_estimate`, `spec_estimate`, `malformed_task_lines`,
`clarify_sessions`, `clarify_session_bullets`. The two uncovered
functions are:

- `parser::_has_checked_tasks` ‚Äî leading-underscore internal helper
  reached transitively from `lifecycle_phase`'s "implementing ‚Üí
  analyzing" branch (covered in the existing `lifecycle_phase`
  cases). Genuinely covered, just not named.
- `parser::_has_clarifications_section` ‚Äî same shape. Internal
  predicate consumed by `clarify_sessions`, which has happy-path and
  no-sessions cases in the suite (lines 97-98).

Effective coverage is 12/12. No follow-up needed.

### `config.sh` ‚Äî 6 / 14 functions, 43% (genuine gap)

`tests/unit/config.bats` (363 lines) covers the public getters
(`get_team_id`, `get_project_id`, `get_workflow_state_uuid`,
`get_default_state_uuid`), plus `load` and `validate`. Uncovered:

- `config::_die`, `config::_warn` ‚Äî internal logging helpers.
  Reached transitively whenever `validate` flags a malformed UUID
  (the `malformed UUID is flagged` test does invoke `_die` via the
  exit-2 path). Genuinely covered, just not named.
- `config::_strip`, `config::_parse_file`, `config::_require_loaded`
  ‚Äî internal parser primitives. `_parse_file` is invoked on every
  `config::load` call and covered transitively. `_require_loaded` is
  invoked from every getter and covered transitively. `_strip` is
  the YAML whitespace/quote stripper and is exercised on every
  loaded value.
- `config::get_operator_user_id`, `config::get_operator_name`,
  `config::get_operator_email` ‚Äî these three operator getters are
  genuinely not unit-tested. The integration suite exercises them
  via `src/install.sh` and `src/reconcile.sh`'s assignee resolution
  (`tests/integration/us4-seed-*.bats`, `us1-fresh-reconcile.bats`),
  but no `config.bats` case asserts that
  `config::get_operator_email "config-fixture.yml"` returns the
  fixture value. **This is the one real unit-test gap in
  `config.sh`.** Low-risk because the three getters share the
  identical lookup shape as `get_team_id` / `get_project_id`, but a
  direct fixture-driven assertion would be cheap.

### `graphql.sh` ‚Äî 2 / 10 functions, 20% (largely justified)

`tests/unit/graphql.bats` (337 lines) covers the public `query` and
`mutate` entry points. Uncovered:

- `graphql::_load_api_key` ‚Äî reads `LINEAR_API_KEY` from env or
  `.env`. Has env-set, env-empty, and `.env`-fallback branches.
  Currently exercised only transitively (the bats suite stubs the
  key directly into the env). A dedicated fixture would tighten
  this.
- `graphql::_post`, `graphql::_request` ‚Äî the curl-wrapper layer.
  Cannot be reached without either a live Linear endpoint or a curl
  stub; the existing tests stub the entire `graphql::_request`
  function. Error paths (non-2xx, rate-limit, network failure) are
  therefore not exercised at the unit layer. Integration test
  coverage is also limited because the integration suite stubs
  `graphql::query`/`mutate` rather than `_post`/`_request`. **This
  is the principal unit-test gap.** A curl-faking shim (`curl=()
  { ... }` in bats) would let us hit the 5xx-retry, 429-rate-limit,
  and JSON-parse-error branches.
- `graphql::_check_rate_limit`, `graphql::_validate_variables`,
  `graphql::_build_operation` ‚Äî internal helpers reached
  transitively by every covered `query`/`mutate` case. Genuinely
  covered, just not named.
- `graphql::_log_error`, `graphql::_log_warn` ‚Äî diagnostic
  emitters. Reached by every error-path test in `graphql.bats` and
  by `config.sh`'s malformed-UUID test that exits 2. Genuinely
  covered, just not named.

Effective unit coverage is ~70% (8 of 10 functions touched
transitively); the dedicated gap is the curl-boundary error matrix.

### `install.sh` ‚Äî 0 / 48 functions, 0% (integration only ‚Äî by design)

No unit tests; 48 functions across 2 087 LOC. Module is a sequential
operator workflow (interactive prompts, MCP-OAuth detection, hook
installation, optional seed-inline, optional Action installation) ‚Äî
the public contract is "run end-to-end against a real `.specify/`
tree and emerge with a working bridge", which doesn't decompose into
isolated unit tests cleanly. Coverage of the install path is
provided by `tests/integration/us4-install-action.bats`,
`us4-seed-fresh-workspace.bats`, `us4-seed-idempotent.bats`,
`us4-seed-prompt.bats`, `us4-unseeded-halts.bats` ‚Äî five end-to-end
scenarios that script-invoke `src/install.sh` against a temporary
worktree. Recommended unit-test slices (see follow-ups) would target
the pure-function helpers: `_version_ge`, `_substitute_uuid_placeholder`,
`_substitute_operator_field`, `_render_hook_block`,
`_hook_already_registered`. Those five together would lift `install.sh`
to a meaningful ~30% direct unit coverage without crossing the
subprocess boundary.

### `seed.sh` ‚Äî 0 / 18 functions, 0% (integration only ‚Äî by design)

964 LOC. Same posture as `install.sh`: every public function talks
to Linear via `graphql::mutate` (workflow-state create, label
create, team-states query). Unit-testing requires a GraphQL stub.
Covered by `tests/integration/us4-seed-fresh-workspace.bats`,
`us4-seed-idempotent.bats`, `us4-seed-prompt.bats` ‚Äî three scenarios
that drive the seed flow against a mocked GraphQL endpoint.
Function-level unit candidates: `render_workflow_uuid_block`,
`render_default_state_uuid_block`, `write_config_uuids`,
`capture_default_states` ‚Äî these four are pure-string-manipulation
helpers and would lift `seed.sh` to ~22% direct unit coverage.

### `status.sh` ‚Äî 0 / 14 functions, 0% (integration only ‚Äî by design)

884 LOC. Pure read-only Linear-vs-filesystem drift inspector.
Covered by `tests/integration/us3-status-staleness.bats`. Function
candidates for unit slicing: `compute_completion`, `compute_drift`,
`extract_memory_branch`, `emit_json`, `_supports_color`. These are
deterministic given fixture inputs and would be cheap to unit-test ‚Äî
~36% direct coverage.

### `pull.sh` ‚Äî 0 / 16 functions, 0% (integration only ‚Äî by design)

866 LOC. Cross-repo unified spec view, Linear-anchored. Same posture
as `status.sh` ‚Äî read-only, GraphQL-driven. Covered by
`tests/integration/us3-pull-cross-repo.bats`. Function candidates
for unit slicing: `build_filter_json`, `extract_memory_field`,
`feature_number_from_labels`, `phase_label_from_labels`,
`build_issue_url`. Pure parsers, easy to fixture-test.

### `reconcile.sh` ‚Äî 0 / 41 functions, 0% (integration only ‚Äî by design)

2 607 LOC. The bridge's spine: enumerates specs, queries Linear,
diffs, mutates. Covered by 11 of the 16 integration scenarios
(`us1-*`, `us2-*`, `retroactive.bats`). Critical pure helpers that
WOULD unit-test cleanly include `compose_issue_description` (the
canonical-rebuild function refactored under Plan B / commit
`2979d96`), `compose_subissue_checklist`, `_extract_overview`,
`_strip_fence`, `render_memory_block`, `render_overview_block`,
`render_diagrams_block`, `json_string`, `json_array`,
`_idle_window_days`, `_lifecycle_is_started`,
`_desired_project_state`, `subissue_state_key`. Thirteen
pure-string/pure-logic helpers across the description-render and
project-status layers ‚Äî these are the highest-leverage unit-test
backfill targets for post-v0.1.0, lifting `reconcile.sh` to
~32% direct unit coverage and protecting the Principle II canonical
rebuild against future regressions.

## Verdict

**Cleared for v0.1.0.** The unit-suite headline (~11% LOC, ~20%
function-level across the whole `src/` tree) understates the true
quality posture because (a) 100% of git_helpers, ‚â•83% of parser
and summary, and effective ~70% of graphql and config are covered
where unit testing is the right tool, and (b) the five large
GraphQL-talking modules (install, seed, status, pull, reconcile ‚Äî
67% of `src/` LOC) are validated by 16 end-to-end integration
scenarios that exercise their public contract against mocked
endpoints. The four-module pure-logic core (config, git_helpers,
parser, summary ‚Äî the modules whose bugs would surface as data-
shape regressions rather than RPC failures) sits at an effective
~80% function coverage, which is the right place for v0.1.0
investment. The principle-conformance audit at
`validation/constitution-recheck-2026-05-28.md` independently
confirms the integration layer's behavioural contract; coverage
numbers are a complement to that audit, not a substitute.

The one unit-test gap that would justify a follow-up before a
hypothetical v0.2.0 dogfood expansion is the `graphql.sh`
curl-boundary error matrix (5xx-retry, 429-rate-limit, JSON-parse).
That gap is currently masked by the integration tests stubbing
above the failure surface; if the curl layer regressed, neither
unit nor integration tests would catch it.

## Recommended follow-ups (post-v0.1.0, non-blocking)

1. **`graphql.sh` curl-boundary unit slice.** Add a bats fixture
   that defines `curl() { ... }` and asserts the retry / rate-limit
   / JSON-parse-error branches of `graphql::_post` and
   `graphql::_request`. Lifts `graphql.sh` from 20% direct to ~70%
   direct coverage and closes the only "real regression could go
   undetected" gap surfaced by this audit.
2. **`config.sh` operator-getter assertions.** Add three
   one-liner fixture-driven cases for `get_operator_user_id`,
   `get_operator_name`, `get_operator_email`. Lifts `config.sh`
   from 43% direct to 64% direct coverage; ~5 LOC of test code.
3. **`reconcile.sh` pure-helper slice.** Add a
   `tests/unit/reconcile.bats` that covers the 13 pure-logic
   helpers enumerated above (`compose_*`, `render_*`, `_extract_*`,
   `_strip_*`, `json_*`, `_idle_window_*`, `_lifecycle_*`,
   `_desired_project_state`, `subissue_state_key`). Lifts
   `reconcile.sh` from 0% to ~32% direct coverage and pins the
   Plan B canonical-rebuild contract.
4. **`install.sh` pure-helper slice.** Add unit cases for
   `_version_ge`, `_substitute_uuid_placeholder`,
   `_substitute_operator_field`, `_render_hook_block`,
   `_hook_already_registered`. Lifts `install.sh` from 0% to ~10%
   direct coverage.
5. **`status.sh` / `pull.sh` parser slice.** Symmetric to the
   `reconcile.sh` slice ‚Äî `compute_completion`, `compute_drift`,
   `build_filter_json`, `extract_memory_field`,
   `feature_number_from_labels`, `phase_label_from_labels`. Lifts
   both from 0% to ~30% direct coverage.
6. **Switch to a kcov-friendly bats runner in CI.** Either pin to
   a bats-core release that resolves the PS4 mirror loop, or run
   the unit suite under a `bash --norc -c 'source tests/unit/X.bats'`
   shim that bypasses bats's parallel runner for coverage runs
   only. Once that lands, replace the manual estimate in this file
   with kcov line numbers.

None of (1)-(5) are constitutional blockers and none are required
for v0.1.0 ship. (6) is infrastructure rather than coverage and
can wait until after the first community-bridge install lands.
